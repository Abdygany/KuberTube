# CLAUDE.md — Learnspace

Этот файл — операционный контракт для Claude Code в репозитории Learnspace.
Читается автоматически в каждой сессии. Источник истины по продукту — `PROJECT.pdf`
в корне репозитория; этот файл фиксирует, **как** работать, а не **что** строить.

---

## 0. Что такое Learnspace (one-liner для контекста)

Учебный workspace для студентов 18-25 лет, защищающий намерение учиться от
рекомендаций, лент и алгоритмических отвлечений. Стек — TypeScript-монорепо:
Next.js 15 (App Router) + Hono + tRPC + Drizzle + Postgres, BYO API-ключи
(YouTube Data v3, Brave Search, Anthropic — опционально), деплой Vercel + Railway,
шифрование ключей AES-256-GCM. Фазы разработки 0→5 описаны в PROJECT.pdf §6.

## 1. Четыре принципа поведения (адаптация Karpathy)

### 1.1 Думай до кода

- Сформулируй задачу как проверяемую цель **до** первого Edit/Write.
- Если в запросе есть неоднозначность — назови её и спроси, не выбирай молча.
- Озвучь допущения, на которых строишь решение (например, «считаю, что фильтры
  применяются на стороне провайдера, а не пост-фильтрацией в БД»).
- Когда не понимаешь — скажи «не понимаю X», не маскируй уверенным тоном.

### 1.2 Минимум кода

- Пиши столько, сколько нужно для запрошенного. 50 строк лучше 200, если
  задача решается.
- Никаких спекулятивных абстракций «на будущее». Три похожих места — лучше
  одного преждевременного хелпера.
- Никакой defensive-валидации внутренних вызовов. Валидируй только границы:
  ввод пользователя, ответы YouTube/Brave/Anthropic, парсинг Readability.
- Никаких feature-флагов и backwards-compat прослоек, если можно просто
  изменить код.

### 1.3 Хирургические правки

- Меняй ровно то, что просили. Не «улучшай» соседние строки, имена,
  форматирование, комментарии.
- Не удаляй существующий мёртвый код, если про него не спросили.
- Удаляй только то, что осиротело **из-за твоих** правок.
- Стиль уже выбран командой (см. конфиги ESLint/Prettier) — следуй ему, не
  навязывай новый.

### 1.4 Цель → проверка

- Перед началом задачи сформулируй критерий «готово» одной фразой.
- Для UI-изменений критерий проверяется в браузере на dev-сервере (golden path
  - edge case + регрессия в соседней фиче). Type-check и тесты подтверждают
    корректность кода, но не корректность фичи.
- Если проверить не можешь — скажи это явно, не выдавай за выполненное.

## 2. Девять UX-принципов Learnspace (закон продукта)

Эти принципы — закон. К ним возвращайся при **каждом** сомнительном решении в
коде или дизайне. Полный текст — PROJECT.pdf §5.

1. Никаких рекомендаций «вам может быть интересно».
2. Никакого автоплея.
3. Никаких бесконечных лент. Списки имеют конец.
4. Никаких ярких уведомлений ради привлечения.
5. Никаких авто-пушей про «активность» и «стрики».
6. Никаких стриков, бейджей, очков опыта.
7. Понятный быстрый escape без «вы уверены?» каждый раз.
8. Без dark patterns. Кнопки удаления одного цвета с подтверждениями.
9. Закрытая система: пользователь видит только то, что сам положил в
   workspace. Никаких подмешанных «похожих» или «рекомендуемых».

Если PR/коммит нарушает любой из девяти — стоп, обсуди с человеком.

## 3. Жёсткие технические инварианты

Не нарушай без явного решения, обновляющего PROJECT.pdf:

- **Контент не наш — не храним.** Только URL, метаданные, заметки. Тело статей
  после Readability не сохраняется в БД.
- **API-ключи — только AES-256-GCM, master key из `ENCRYPTION_KEY` env.**
  Никогда не логируй plaintext-ключ. Никогда не складывай ключ в JSON-ответ.
- **Soft delete + cron физического удаления через 30 дней** для workspaces,
  resources, notes. Не делай hard delete напрямую.
- **YouTube IFrame:** `rel=0`, `modestbranding=1`, `iv_load_policy=3`,
  `enablejsapi=1`, `ended` → `stopVideo()`. Никакого автоплея, никакой сетки
  рекомендаций.
- **Поиск:** параллельные запросы YouTube + Brave, фильтры на стороне
  провайдера, кэш `search_results_cache` TTL 24 часа, без своего ранжирования
  (порядок от источника).
- **tRPC** — единственный транспорт фронт↔бэк. Не добавляй параллельный REST,
  если не обсудили.
- **Гибкие JSON-поля** (`metadata_json`, `settings_json`, `filters_json`) — для
  эволюции без миграций. Не размазывай новые поля по отдельным колонкам, если
  не нужен индекс.

## 4. Стек и команды

```
apps/web   — Next.js 15 (App Router, RSC, SSR landing/docs)
apps/api   — Hono + tRPC backend
packages/db        — Drizzle schema + миграции
packages/api-types — общие типы tRPC роутеров
packages/core      — бизнес-логика (поиск, фильтрация, парсинг)
packages/ui        — shadcn/ui компоненты
packages/config    — eslint/tsconfig/tailwind пресеты
```

Менеджер — `pnpm`, оркестрация — `turbo`. Не вводи `npm`/`yarn`/`bun` параллельно.

## 5. Скиллы в этом репозитории

В `.claude/skills/` лежат четыре скилла, расширяющие поведение Claude Code под
Learnspace. Каждый — отдельная папка с `SKILL.md`-frontmatter, который Claude
загружает по триггеру.

| Скилл                 | Когда срабатывает                                                                                                    |
| --------------------- | -------------------------------------------------------------------------------------------------------------------- |
| `karpathy-guidelines` | Перед любым кодированием — дисциплина из §1 в развёрнутой форме с примерами.                                         |
| `learnspace-soul`     | Когда нужно писать пользовательские тексты, копию, доку, README, landing — даёт голос продукта.                      |
| `gstack-roles`        | Когда задача требует одной из 8 ролей (CEO/Eng Manager/Designer/Staff/QA/Release/Security/DX). Слэш-команды `/ls-*`. |
| `superpowers`         | Для нетривиальных задач: `/brainstorm` → `/write-plan` → `/execute-plan` → `/finish`.                                |

Загружай скилл, когда совпал триггер из его frontmatter `description`. Не
дублируй вручную то, что скилл и так делает.

## 6. Git и коммиты

- Активная ветка разработки — `claude/add-ai-agent-skills-II9Tk` (для текущей
  работы по скиллам). Дальше — feature-ветки от `main`.
- Коммиты — императивные, по-русски или по-английски, но единообразно в рамках
  ветки. Объясняют **почему**, не **что**.
- Не амендь опубликованные коммиты. Не используй `--no-verify`. Не делай
  `git push --force` в `main`.
- PR создавай **только** по явной просьбе пользователя.

---

## 7. Чего не делать никогда

- Не добавлять рекомендации, автоплей, бесконечные ленты, стрики, бейджи —
  даже «временно для теста».
- Не вводить аналитику обучения / геймификацию / push-уведомления о «вашей
  активности».
- Не интегрировать книги, подкасты, академические PDF, мобилку, командную
  работу, биллинг — это явный Won't have for MVP (PROJECT.pdf §2).
- Не выдумывать URL-ы пакетов, версии, API-эндпоинты. Если не уверен —
  проверь в репозитории/документации.
- Не запускать длинные миграции/деструктивные команды без подтверждения.
