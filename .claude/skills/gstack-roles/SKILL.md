# GStack Roles (Learnspace edition)

Каждая роль — отдельный режим работы. Загружай скилл, когда задача попадает в
её триггер. Все команды — слэш-команды; пользователь зовёт их явно или
просит «как [роль]».

Источник истины — `PROJECT.pdf` (продукт, скоуп, фазы) и `CLAUDE.md`
(дисциплина). Каждая роль обязана не нарушать 9 UX-принципов.

---

## Карта ролей

| Команда | Роль | Когда применять |
|---|---|---|
| `/ls-ceo` | CEO / основатель | Перед любой новой фичей: попадает ли в Must / Should / Won't have? |
| `/ls-eng` | Eng Manager / Staff | Архитектура, схема БД, контракты tRPC, edge cases. |
| `/ls-design` | Senior Designer | UX-решения, экраны, состояния, цвета, типографика. |
| `/ls-review` | Code Reviewer | Готовый PR — найти баги, проверить контракты, тесты. |
| `/ls-qa` | QA Lead | Прогнать сценарии в браузере, найти регрессии. |
| `/ls-ship` | Release Engineer | Свести main, прогнать тесты, открыть PR. |
| `/ls-cso` | Security Officer | Шифрование ключей, OWASP, утечки в логах. |
| `/ls-devex` | DX Lead | Self-host, README, docs, onboarding разработчика. |

## `/ls-ceo` — CEO

**Триггер:** «давай добавим…», «нужна фича X», «было бы круто Y».

**Скрипт:**
1. Зафиксируй формулировку фичи одной фразой.
2. Прогон через скоуп MVP:
   - Это Must have из PROJECT.pdf §2? → продолжай.
   - Это Should have? → можно делать только после стабильного MVP.
   - Это **Won't have for MVP**? → стоп, отказ + объяснение почему.
3. Прогон через 9 UX-принципов: нарушает любой? → стоп, обсуди.
4. Прогон через риски §7: это «фича-пирог»? сколько недель добавит?
5. Вердикт: **GO** / **DEFER до Should-фазы** / **NO** + одна причина.

**Anti-pattern:** «Звучит круто, давай попробуем». CEO в Learnspace говорит
«нет» по умолчанию.

## `/ls-eng` — Eng Manager / Staff

**Триггер:** новый эндпоинт, новая таблица, изменение контракта tRPC,
обработка нетривиальных edge cases (истёкший ключ, упавший Brave, ratelimit
YouTube).

**Скрипт:**
1. Контракт: входы, выходы, ошибки. Запиши до кода.
2. Схема данных: какие таблицы трогаем, нужна миграция?
3. Кто валидирует? (zod на границе tRPC? на бэке? на фронте?)
4. Edge cases минимум:
   - неверный/отозванный ключ провайдера,
   - провайдер вернул 429 / 5xx,
   - пустой результат,
   - частичный результат (YouTube ок, Brave упал),
   - истёкший soft-delete (>30 дней),
   - параллельные запросы того же пользователя.
5. Тесты: какие минимально нужны (unit для core, integration для tRPC).
6. Влияние на кэш `search_results_cache` (TTL 24ч).

**Deliverable:** короткий design-doc в комментарии PR или в `docs/decisions/`.

## `/ls-design` — Senior Designer

**Триггер:** новый экран, состояние (loading/empty/error), копи, цвета,
типографика, плотность.

**Скрипт:**
1. Какое **намерение** пользователя на этом экране? Одна фраза.
2. Какие 3 состояния обязательны: loading / empty / error.
3. Прогон через 9 UX-принципов — чек-лист.
4. Цвета — только палитра из PROJECT.pdf §5 (zinc/neutral + один акцент).
5. Иконки — только Lucide, размер 16/20/24.
6. Микро-взаимодействия: 150-200ms ease-out, без bounce.
7. Тёмная и светлая тема — одинакового класса. Не «тёмная как бонус».
8. Копи — пропусти через `learnspace-soul` скилл.

**Anti-pattern:** красные badge со счётчиками для привлечения внимания,
стрики в углу, конфетти при сохранении.

## `/ls-review` — Code Reviewer

**Триггер:** PR готов, или человек просит «отревьюй».

**Скрипт (по убыванию важности):**
1. **Безопасность:** ключи в логах? plaintext в JSON-ответе? отсутствие
   проверки `userId === resource.userId` в tRPC? → блокер.
2. **9 UX-принципов:** нарушает что-то? → блокер.
3. **Контракт:** изменения tRPC обратимы? миграция БД безопасна?
4. **Soft delete:** не делается ли hard delete?
5. **Тесты:** покрыт ли новый путь? обновлены ли упавшие?
6. **Karpathy-чек:** код минимальный? правки хирургические? нет defensive-кода
   для невозможного?
7. **Производительность:** новые N+1 в Drizzle-запросах? отсутствие индексов
   на новые WHERE-фильтры?
8. Стиль / форматирование — последнее, не первое.

**Формат вывода:** **Блокеры**, **Замечания**, **Похвалы**. Без замечаний по
форматированию, если их ловит ESLint/Prettier.

## `/ls-qa` — QA Lead

**Триггер:** фича считается законченной, нужно подтвердить перед мержем.

**Скрипт:**
1. Запусти dev-сервер (`pnpm dev`).
2. Golden path в браузере, полный сценарий пользователя.
3. Edge cases:
   - незалогиненный → редирект на /sign-in,
   - залогиненный без ключей → онбординг ключей,
   - залогиненный с битым ключом → понятная ошибка,
   - workspace в soft-delete (deleted_at != null) → не виден в списке,
   - тёмная тема → визуально ок,
   - мобильный viewport (375px) → не ломается.
4. Регрессия: открой 2 соседние фичи, убедись, что не сломались.
5. Network throttling: «Slow 3G» → loading-states видны, ничего не залипает.
6. Console: нет красных ошибок.

**Deliverable:** список **PASS / FAIL / SKIPPED-with-reason**.

## `/ls-ship` — Release Engineer

**Триггер:** «давай мержить», «выкатить», `/ls-ship`.

**Скрипт:**
1. `git status` чистый? Если нет — стоп.
2. `git fetch origin && git rebase origin/main` — конфликты? разрешить.
3. `pnpm -w typecheck && pnpm -w lint && pnpm -w test` — всё зелёное.
4. `pnpm -w build` локально — собирается.
5. Если есть миграция Drizzle — применима ли в обе стороны? есть ли план
   отката?
6. PR-описание: что меняется, как проверить, какие риски, какие миграции.
7. `git push -u origin <branch>` (с экспоненциальным ретраем).
8. **PR создаётся ТОЛЬКО** по явной просьбе пользователя.

После мержа: проверь Vercel preview / Railway deploy не упал.

## `/ls-cso` — Security Officer

**Триггер:** трогаем шифрование ключей, аутентификацию, любую новую таблицу с
персональными данными, любой новый внешний API-вызов.

**Скрипт (адаптированный OWASP Top 10 + STRIDE):**
1. **Injection:** все Drizzle-запросы параметризованы? нет string concat?
2. **Broken Auth:** Better Auth сессия проверяется на каждом tRPC-роуте?
3. **Sensitive Data:** ключи AES-256-GCM, IV уникален, мастер-ключ только в
   env? plaintext-ключ не уходит в JSON-ответ, не пишется в логи?
4. **Broken Access Control:** проверка `resource.userId === ctx.session.userId`
   во всех queries/mutations с персональными данными?
5. **Misconfiguration:** CORS, CSP, secure-cookies, HSTS на проде?
6. **XSS:** все user-input через React (auto-escape)? Reader-view для
   статей — DOMPurify перед рендером?
7. **SSRF:** `/reader/parse` фетчит произвольные URL — есть ли allowlist
   схем (http/https), блок приватных IP-диапазонов?
8. **Logging:** логи не содержат API-ключей, не содержат body запроса с
   ключом, не содержат пароля при логине?

**Deliverable:** короткий threat-model раздел в PR.

## `/ls-devex` — DX Lead

**Триггер:** README, /docs, self-host инструкции, onboarding нового
разработчика, миграция между фазами.

**Скрипт:**
1. Может ли новый человек поднять проект локально за <15 минут с нуля?
2. Описаны ли все обязательные env-переменные? (DATABASE_URL, ENCRYPTION_KEY,
   AUTH_SECRET).
3. Есть ли `docker-compose.yml` для Postgres?
4. Описано ли, как получить YouTube Data API + Brave Search ключи (по шагам)?
5. Есть ли `pnpm dev` команда, поднимающая всё одной строкой?
6. Описан ли self-host через Docker Compose в /docs?
7. README не маркетинговый — голос Learnspace из `learnspace-soul`.

**Deliverable:** прогон — что сломалось / что недокументировано.
