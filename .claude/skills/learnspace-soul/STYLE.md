# STYLE.md — как Learnspace говорит

## Базовые правила

- **Короткие предложения.** Если фраза не помещается в строку — режь.
- **Глагол вперёд** в кнопках: «Создать workspace», не «Создание workspace».
- **«Вы»** — нейтральное, не «ты», не «друг», не «коллега».
- **Никаких восклицательных знаков** в продукте. Точка достаточна.
- **Цифры цифрами:** «3 ресурса», не «три ресурса».
- **Никаких эмодзи** в продуктовом UI (для иконок есть Lucide).

## Микро-копи: примеры

### Кнопки

| Контекст | OK | Не OK |
|---|---|---|
| Создать workspace | Создать workspace | Поехали! / Начать обучение |
| Подтвердить удаление | Удалить | Да, удалить навсегда |
| Закрыть видео | Готово | Закончил просмотр |
| Сохранить заметку | Авто-сохранение видно индикатором | «Сохранено!» с восклицанием |

### Пустые состояния

| Экран | OK |
|---|---|
| Нет workspace'ов | «Создайте первый workspace, чтобы начать.» |
| Пустой workspace | «В этом workspace пока нет ресурсов. Запустите поиск, чтобы добавить.» |
| Поиск ничего не нашёл | «Ничего не найдено по этим фильтрам. Попробуйте смягчить актуальность или баланс источников.» |

### Ошибки

Структура: `[что произошло]. [что сделать].`

| OK | Не OK |
|---|---|
| «Ключ YouTube не валиден. Проверьте его в Настройках → API-ключи.» | «Упс! Что-то пошло не так :(» |
| «Brave Search не отвечает. Попробуйте через минуту.» | «Извините, мы работаем над этим!» |
| «Не удалось разобрать статью. Откройте оригинал.» | «К сожалению, технические сложности.» |

### Подтверждения деструктивных действий

- Один экран. Одна кнопка действия. Одна кнопка отмены.
- Кнопка «Удалить» **того же визуального веса**, что и «Отмена».
- Никаких красных модалок с восклицательным знаком.
- Текст: «Workspace будет перемещён в корзину. Восстановить можно в течение
  30 дней.»

## Onboarding-тон

4-5 экранов: что за продукт → уровень → длительность → баланс источников →
актуальность.

- Каждый экран = один вопрос или одно объяснение.
- Можно пропустить любой шаг.
- Нет прогресс-бара с конфетти.
- Нет «Молодец!» между шагами.
- Кнопка «Пропустить» того же веса, что «Далее».

## Документация

- Шаги нумерованные.
- Скриншоты только когда без них непонятно.
- Никаких «Привет, друг!» или «Добро пожаловать в удивительный мир…».
- Начинай с задачи пользователя, не с истории продукта.

## Landing

Структура: `Проблема → Решение → Как это работает → FAQ`.

- Hero — одна фраза-обещание + одна кнопка.
- Без видео-фонов, параллакса, анимированных градиентов.
- Без счётчика звёзд GitHub в hero (можно в footer).
- Без «Trusted by 10,000+ users» — нечестно для MVP.

## Чек-лист перед публикацией текста

1. Прошёл бы текст без правок в README хорошего open-source проекта?
2. Есть ли восклицательные знаки? Удали.
3. Есть ли эмодзи? Удали (кроме явной просьбы пользователя).
4. Есть ли «Упс», «Ой», «К сожалению»? Перепиши без них.
5. Защищает ли текст внимание пользователя или забирает его?
