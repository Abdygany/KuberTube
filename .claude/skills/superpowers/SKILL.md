# Superpowers (Learnspace edition)

Этот скилл — каркас методологии для нетривиальных задач. Маленькие правки
делай напрямую под `karpathy-guidelines`. Этот скилл нужен, когда задача:

- трогает больше одной папки (`apps/` × `packages/`),
- требует миграции БД,
- требует решения «как делать», а не только «что делать»,
- занимает больше двух часов работы.

Четыре фазы строго последовательные. Не пропускай.

---

## Фаза 1. `/ls-brainstorm`

**Цель:** превратить смутную идею в чёткую формулировку, отбросив очевидно
плохие варианты.

**Скрипт:**

1. Перепиши запрос пользователя как **проблему**, не как решение.
   - Плохо: «добавить кэш Redis».
   - Хорошо: «повторные запросы поиска по той же теме медленные и сжигают
     квоту YouTube».
2. Назови минимум 2 альтернативы решения.
3. Для каждой альтернативы:
   - Что выиграем,
   - Что потеряем,
   - Сколько примерно стоит (часы / недели),
   - На какую фазу PROJECT.pdf попадает.
4. Прогон через 9 UX-принципов: какие альтернативы отпадают?
5. Прогон через скоуп MVP: какие отпадают как Won't have / Should have?
6. Рекомендация — **одна** альтернатива, одна причина.

**Когда останавливаться:** когда у пользователя есть один вариант, в котором
он уверен. Не уходи в реализацию, пока не услышал «да, делаем X».

**Anti-pattern:** «Я подумал, и решил пойти путём А». Решает пользователь,
скилл показывает варианты.

## Фаза 2. `/ls-plan`

**Цель:** превратить выбранное решение в проверяемый план.

**Шаблон плана:**

```
## Цель (одна фраза)
…

## Критерий «готово» (как проверим)
…

## Затронутые места
- apps/web: …
- apps/api: …
- packages/db: миграция? да/нет
- packages/core: …

## Контракт изменений
- tRPC: новые/изменённые роуты, входы, выходы, ошибки.
- Схема БД: таблицы, индексы, JSON-поля.
- Внешние API: какие квоты затрагиваем.

## Шаги (атомарные, каждый можно ревьюить отдельно)
1. …
2. …
3. …

## Тесты
- Unit (packages/core): …
- Integration (apps/api): …
- Ручной браузерный (apps/web): …

## Риски
- …

## Что НЕ делаем в этом PR
- …
```

**Правило:** план показываем человеку и ждём «ок» перед `/ls-exec`. Если план
получается длиннее ~25 пунктов — задача слишком большая, режь на два PR.

---

## Фаза 3. `/ls-exec`

**Цель:** выполнить план дисциплинированно, не сползая в скоуп-крип.

**Скрипт:**

1. Открой план рядом. Каждый шаг — отдельный логический блок.
2. После каждого шага:
   - Запусти соответствующий проверяющий цикл (typecheck/lint/тест).
   - Отметь шаг выполненным в TodoWrite.
3. Если по ходу всплывает «о, надо ещё Y»:
   - Y попадает в текущий план? делай.
   - Y не в плане? **запиши в «Что отложено»** и продолжай. Не делай молча.
4. Не трогай файлы, не упомянутые в «Затронутые места», без явного решения.
5. Соблюдай `karpathy-guidelines` принцип 3 (хирургические правки).

**Стоп-условия — спроси человека:**

- План оказался неверным (выяснилась критичная неучтённая деталь).
- Появилась миграция БД, которой не было в плане.
- Затронут один из 9 UX-принципов.
- Появилась необходимость менять `ENCRYPTION_KEY` или формат шифрования.

## Фаза 4. `/ls-finish`

**Цель:** убедиться, что задача действительно закрыта, и понять, что дальше.

**Скрипт:**

1. Прогон `/ls-qa` (см. `gstack-roles`): браузерный smoke + регрессия.
2. Прогон `/ls-review` (см. `gstack-roles`): self-review своих изменений.
3. Прогон `/ls-cso`, если задача трогала ключи / auth / новые внешние URL.
4. Обнови README/docs, если изменилось пользовательское поведение.
5. Проверь: критерий «готово» из плана выполнен дословно?
6. Сводка пользователю — две колонки:
   - **Сделано:** список с файл:строка ссылками.
   - **Отложено:** что выяснилось по ходу, но не вошло (с предложением — в
     этот PR следующим шагом, отдельный PR, или Won't have).
7. Спроси: «Мержить через `/ls-ship`, или ещё что-то добавить?»

**Anti-pattern:** «Готово!» без QA-прогона и без явного критерия.

---

## Когда какой скилл подключается (карта)

```
любая нетривиальная задача
        │
        ▼
  /ls-brainstorm  ← варианты, выбор
        │
        ▼
  /ls-plan        ← план, контракт, шаги
        │
        ▼
  /ls-exec        ← код, по шагу за раз
        │
        ▼
  /ls-finish      ← QA + review + сводка
        │
        ▼
  /ls-ship        ← (по просьбе) PR + мерж
```

На всех фазах активны:

- `karpathy-guidelines` — дисциплина кода,
- корневой `CLAUDE.md` — инварианты Learnspace,
- `learnspace-soul` — для любых пользовательских текстов,
- роли из `gstack-roles` — точечно по триггеру.
