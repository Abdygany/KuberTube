# Karpathy Guidelines (Learnspace edition)

Этот скилл — операционная развёртка четырёх принципов из корневого CLAUDE.md.
Загружай его, когда садишься за нетривиальную задачу — особенно если она
затрагивает поиск, видео-плеер, шифрование ключей или схему БД.

---

## Принцип 1. Думай до кода

**Что делать:**
1. Перепиши запрос пользователя одной фразой как **проверяемую цель**.
   - Плохо: «добавить фильтры в поиск».
   - Хорошо: «фильтры `level/duration/balance/freshness` из workspace
     применяются на стороне YouTube + Brave при `POST /workspaces/:id/search`,
     возвращая ≤ 30 карточек, без post-фильтрации в БД».
2. Перечисли явные **допущения** (1-3 пункта).
3. Если допущение критично и непроверяемо без пользователя — **спроси**, не
   выбирай молча.
4. Назови **альтернативные интерпретации**, если их больше одной, и предложи
   ту, которую считаешь правильной.

**Anti-patterns:**
- «Я начну, а по ходу разберёмся» на задаче >30 минут.
- Уверенный тон при неуверенности. Скажи «не знаю X», «не нашёл Y».
- Молча выбрать одну из двух одинаково правдоподобных трактовок.

**Learnspace-checks перед стартом:**
- Не нарушает ли задача один из 9 UX-принципов? Если да — стоп, обсуди.
- Затрагивает ли BYO-ключи или AES-шифрование? Если да — отдельный аудит.
- Влияет ли на soft-delete окно 30 дней? Если да — проверь cron.

## Принцип 2. Минимум кода

**Что делать:**
- Целься в **самую короткую** реализацию, которая закрывает критерий «готово».
- Если написал 200 строк, а можно 50 — перепиши.
- Удаляй любую защиту, которая защищает от **невозможного** в этом контексте.
- Внутренний вызов в монорепо доверяй типам tRPC. Валидируй только границы:
  - вход пользователя (форма, query-params),
  - ответы YouTube Data API / Brave Search / Anthropic,
  - результат `Readability.parse()` (часто падает),
  - расшифровка ключа (может вернуть мусор при неверной ноне).

**Anti-patterns:**
- Хелпер «на будущее» с одним вызывающим.
- Generic-обёртка вокруг fetch с собственной системой ретраев, когда нужен
  один запрос.
- `try/catch` вокруг кода, который физически не бросает.
- Создание `packages/utils` для функции из 4 строк.
- Feature-флаги для кода, который никогда не пойдёт в прод в выключенном виде.

**Правило трёх:** дублирование становится абстракцией только на третьем
повторе. Не на втором.

## Принцип 3. Хирургические правки

**Что делать:**
- Меняй ровно строки, относящиеся к задаче.
- Не реформатируй файл «заодно».
- Не переименовывай переменные «заодно».
- Не добавляй комментарии о том, что код делает, если идентификаторы уже
  говорят сами за себя.
- Не удаляй чужой dead code «заодно» — вынеси в отдельную задачу.

**Что МОЖНО затронуть в той же правке:**
- Строки, которые перестали компилироваться из-за твоих изменений.
- Импорты, осиротевшие из-за твоих изменений.
- Тесты, проверявшие удалённое тобой поведение.

**Anti-patterns:**
- «Refactored while I was here» — нет, это отдельный PR.
- Замена `let` на `const` в файле, где менял одну строку.
- Перенос файла в другую папку «для логики», когда задача — починить баг.

## Принцип 4. Цель → проверка

**Что делать:**
- До старта: одна фраза «фича считается готовой, когда…».
- Для backend: критерий проверяется `curl`-ом или unit/integration тестом.
- Для UI: критерий проверяется в браузере на dev-сервере. Минимум:
  - golden path (типичный сценарий пользователя),
  - 1 edge case (пустой workspace, истёкший ключ, отвалившийся YouTube),
  - 1 регрессионный чек соседней фичи.
- Если запустить и проверить **не можешь** (нет ключей, нет окружения, нет
  доступа к Postgres) — скажи это **явно**. Не выдавай за «готово».

**Чек-лист «готово к репорту»:**
1. Критерий «готово» сформулирован в начале задачи?
2. Type-check (`pnpm -w typecheck`) — зелёный?
3. Lint — зелёный?
4. Тесты, относящиеся к изменению, — зелёные?
5. Если UI — проверил в браузере и записал, что именно проверил?
6. Если не проверил — указал это в репорте отдельной строкой?

## Когда обращаться за человеческим решением

Останови работу и спроси, если:
- Решение нарушает или может нарушить один из 9 UX-принципов.
- Задача требует изменить схему БД (миграция = обсуждение).
- Задача требует трогать `ENCRYPTION_KEY`, формат шифрования или провайдера
  ключей.
- Задача предполагает удаление пользовательских данных в обход soft-delete.
- Сроки и фазы из PROJECT.pdf §6 двигаются.
